# AIStudioBuildWS ↔ moe-cli-proxy-dr 架构诊断

## 0. 精简思维链

```yaml
Observation:
  - BrowserManager 只启动 Camoufox/Playwright BrowserContext，并打开 AI Studio App URL。
  - Flask server 只暴露 7860: /health、控制台、WS 日志开关。
  - WebSocketLogger 只监听浏览器页面内已产生的 websocket 事件，不提供服务端。
  - README 明确写明: AIStudio 页面内通信程序应连接 CLIProxyAPI 的 /v1/ws。
  - README 又声称容器内置 socat: 127.0.0.1:8317 -> cli-proxy-api:8317。
  - 当前 Dockerfile.async 未安装 socat，compose command 也未启动 socat。
Conclusion:
  - 缺的是浏览器页面 JS 到 moe-cli-proxy-dr 的 WS 出站链路，不是 moe-cli-proxy-dr 到本项目 Flask 的入站链路。
  - 当前最大嫌疑: AI Studio App 里配置的是 ws://127.0.0.1:8317/v1/ws，但 aistudio-shard-0 容器内 127.0.0.1:8317 没有转发服务。
```

# 1. 本项目真实架构原理

## 1.1 角色边界

```text
[外部调用方]
   |
   | HTTP/OpenAI/Gemini-compatible API
   v
[moe-cli-proxy-dr]
   - API Gateway
   - 暴露 /v1/models /v1/chat/completions 等接口
   - 暴露 WebSocket Provider 接入端: /v1/ws
   - 等待 AIStudio Provider 浏览器会话主动连入
   |
   | WebSocket 任务分发
   v
[AIStudioBuildWS 容器: aistudio-shard-0]
   - 启动 Camoufox/Playwright
   - 用 Cookie 登录/打开 Google AI Studio App
   - 页面内 JS 通信程序主动连接 moe-cli-proxy-dr 的 /v1/ws
   - 页面内 JS 收到任务后调用 AI Studio/Gemini 页面能力
   - 再通过同一条 WS 把结果回传给 moe-cli-proxy-dr
   |
   | Browser page connection
   v
[Google AI Studio App]
```

## 1.2 BrowserManager 做了什么

`browser/manager.py` 的核心行为：

```yaml
BrowserManager:
  启动:
    - AsyncCamoufox(**launch_options)
  每个账号:
    - browser.new_context()
    - context.add_cookies(cookies)
    - page = context.new_page()
    - page.goto(CAMOUFOX_INSTANCE_URL)
    - WebSocketLogger.attach_to_page(page)
    - handle_successful_navigation(page) 保活
```

它不是任务调度器，也不是 WebSocket Server。

它只是：

```text
浏览器宿主 + Cookie 注入器 + 页面保活器 + WebSocket 旁路日志器
```

真正执行 Gemini 调用/任务接收的逻辑不在 Python 项目内，而是在 `CAMOUFOX_INSTANCE_URL` 指向的 Google AI Studio App 页面内 JS 程序里。

## 1.3 WebSocketLogger 的含义

`browser/ws_logger.py`：

```python
page.on('websocket', self._handle_websocket)
```

这表示：

```text
监听浏览器页面已经发起的 WebSocket 连接
```

不是：

```text
创建一个 WebSocket 服务端供 moe-cli-proxy-dr 连接
```

因此如果日志里没有出现新 WS，说明页面内 JS 没有成功发起连接，或页面没执行到连接逻辑。

# 2. 当前容器端口与服务状态判断

## 2.1 Flask 端口

`main.py` 中：

```python
flask_app.run(host='0.0.0.0', port=7860, debug=False)
```

当 `HG=true` 时，容器会启动 Flask 7860。

当前 `docker-compose.override.yml`：

```yaml
environment:
  - HG=true
```

所以：

```yaml
aistudio-shard-0:
  expected_listen:
    - 0.0.0.0:7860
```

但 `docker-compose.yml` 没有 `ports:`，所以：

```yaml
host_exposed_ports:
  - none
container_internal_ports:
  - 7860
```

这不影响同 Docker network 内其他容器访问 `http://aistudio-shard-0:7860/health`，但不能从宿主直接访问，除非映射端口。

## 2.2 业务 WebSocket 端口

当前项目源码没有以下内容：

```yaml
missing:
  - websocket server route
  - Flask-SocketIO
  - aiohttp/websockets server
  - /v1/ws handler
  - 8317 listener
```

所以：

```yaml
aistudio-shard-0:
  expected_business_ws_server: false
```

## 2.3 关键缺口：README 声称有 socat，但 async 路径没有

README 说：

```text
容器内置 socat 端口转发:
127.0.0.1:8317 -> cli-proxy-api:8317
```

但当前事实：

```yaml
Dockerfile:
  installs:
    - socat

Dockerfile.async:
  installs:
    - no socat

docker-compose.override.yml:
  image: aistudio-websocket:async

compose command:
  - camoufox fetch ...
  - python main.py
  - no socat startup
```

因此当前运行的 async 镜像大概率没有 socat，也没有启动 8317 转发。

这会导致：

```text
AI Studio 页面内 JS 如果配置为 ws://127.0.0.1:8317/v1/ws
  -> 在浏览器所在容器网络命名空间内访问 127.0.0.1:8317
  -> 127.0.0.1 是 aistudio-shard-0 自身
  -> 没有服务监听 8317
  -> WebSocket 连接失败
  -> moe-cli-proxy-dr 看不到 provider connected
  -> gemini-3-flash-preview 不可用
```

# 3. 对三个问题的精准回答

## Q1: BrowserManager 开启浏览器后，外部 moe-cli-proxy-dr 怎么把任务分发给这个浏览器？

不是 `moe-cli-proxy-dr` 主动连接 `aistudio-shard-0`。

正确方向是：

```text
AIStudio 页面内 JS 主动连接 moe-cli-proxy-dr 的 WebSocket Server。
```

链路：

```text
BrowserManager
  -> 打开 CAMOUFOX_INSTANCE_URL
  -> Google AI Studio App 页面加载
  -> 页面内预置/配置的 JS 读取 WEBSOCKET_PROXY_URL
  -> JS 发起:
       ws://127.0.0.1:8317/v1/ws
       或 wss://gateway.example.com/v1/ws
  -> moe-cli-proxy-dr 接收 provider 注册
  -> moe-cli-proxy-dr 后续把请求经该 WS 发给页面
```

所以 `BrowserManager` 不直接参与任务分发，只维持页面活着。

## Q2: aistudio-shard-0 现在暴露了端口吗？是不是缺少服务端逻辑或端口没暴露？

分两类：

```yaml
Flask管理端口:
  container_listen: 7860
  host_ports: 未映射
  purpose: health/ui/log toggle
  relation_to_gemini_calls: none

业务WebSocket端口:
  container_listen: 当前应为 none
  required_by_correct_architecture: 不要求本项目监听业务WS
  required_by_local-loopback方案: 需要 socat 监听 127.0.0.1:8317
```

结论：

```yaml
不是缺少 cliproxy-dr 连接进来的 WebSocket Server。
真正缺口是:
  - 页面内连接目标如果是 127.0.0.1:8317，则容器内必须有 socat/转发器监听 8317。
  - 当前 async 镜像/启动命令没有提供这个转发器。
```

## Q3: 如果本项目只是 Flask 服务，为什么没有启动 cliproxy-dr 预期的 WebSocket Server？

因为这个项目原设计不是 `cliproxy-dr backend server`，而是 `AIStudio provider browser worker`。

协议边界：

```yaml
moe-cli-proxy-dr:
  owns:
    - API Gateway
    - /v1/ws WebSocket Server
    - provider registry
    - request queue
    - model routing

AIStudioBuildWS:
  owns:
    - browser lifecycle
    - cookie/session
    - AI Studio page keepalive
    - page websocket visibility/logging
```

如果在 AIStudioBuildWS 里再实现一个 `/v1/ws` WebSocket Server，会变成错误架构：

```text
外部请求 -> moe-cli-proxy-dr -> aistudio-shard-0 WS server -> ??? -> Browser page
```

这中间还缺 Python 到页面 JS 的桥接协议，等于重写 CLIProxyAPI Provider 协议，ROI 很低。

# 4. 根因候选排序

## H1 高概率：容器内 127.0.0.1:8317 没有 socat 转发

证据：

```yaml
README:
  says: ws://127.0.0.1:8317/v1/ws + socat转发

Dockerfile.async:
  lacks: socat

docker-compose.override.yml:
  uses: aistudio-websocket:async

command:
  lacks: socat TCP-LISTEN
```

影响：

```text
页面内 JS 连接 ws://127.0.0.1:8317/v1/ws 必失败。
```

## H2 中概率：AI Studio App 内 WEBSOCKET_PROXY_URL 配错

可能值：

```yaml
bad:
  - ws://cliproxy-dr:8317/v1/ws
  - ws://moe-cli-proxy-dr:8317/v1/ws
  - http://...
  - missing /v1/ws
  - stale old host
  - wrong JWT_TOKEN / wsauth

safe_for_mixed_content:
  - ws://127.0.0.1:8317/v1/ws with socat
  - wss://public-domain/v1/ws
```

注意：在 HTTPS 页面中连接 `ws://非localhost` 可能被 mixed content 阻断。`ws://127.0.0.1` 通常更容易绕过本地回环安全例外，所以 README 才设计 socat。

## H3 中概率：服务名漂移

当前项目引用混乱：

```yaml
seen_names:
  - cliproxy-dr
  - cli-proxy-dr
  - moe-cli-proxy-dr
  - moe-cli-proxy-api
  - cli-proxy-api
```

`docker-compose.override.yml` 的 `NO_PROXY` 包含：

```text
cliproxy-dr,moe-cli-proxy-dr,moe-cli-proxy-api
```

但测试代码 fallback 用：

```text
cli-proxy-dr
```

README 用：

```text
cli-proxy-api
```

如果 socat 目标写错服务名，也会失败。

## H4 低概率：aistudio-shard-0 需要对外暴露 7860

7860 只用于健康检查和 UI，不是业务链路关键点。`moe-cli-proxy-dr` 不应依赖连接 `aistudio-shard-0:7860` 来完成 Gemini 调用。

# 5. 修补方案：最小正确路径

## 5.1 方案 A：保留 README 的 loopback+socat 架构

这是最低风险修复。

### 目标拓扑

```text
AI Studio 页面 JS
  -> ws://127.0.0.1:8317/v1/ws
  -> 容器内 socat 监听 127.0.0.1:8317
  -> 转发到 moe-cli-proxy-dr:8317
  -> moe-cli-proxy-dr /v1/ws
```

### 需要改动

```yaml
Dockerfile.async:
  add apt package:
    - socat

docker-compose.override.yml or command:
  before python main.py:
    - start socat:
      socat TCP-LISTEN:8317,fork,reuseaddr TCP:moe-cli-proxy-dr:8317 &

NO_PROXY:
  include:
    - localhost
    - 127.0.0.1
    - moe-cli-proxy-dr
    - cli-proxy-dr
    - cliproxy-dr

AI Studio App config:
  WEBSOCKET_PROXY_URL:
    - ws://127.0.0.1:8317/v1/ws
  JWT_TOKEN:
    - equals moe-cli-proxy-dr api-key if wsauth=true
```

### 不要做

```yaml
do_not:
  - 在 Flask 上实现 /v1/ws
  - 让 moe-cli-proxy-dr 连接 aistudio-shard-0:7860
  - 把浏览器页面内 URL 改成 ws://moe-cli-proxy-dr:8317，除非确认 Mixed Content 不拦截
```

## 5.2 方案 B：改为 wss 公网/内网 TLS Gateway

适用于不想在容器内 socat 的情况。

```yaml
AI Studio App:
  WEBSOCKET_PROXY_URL: wss://<gateway-domain>/v1/ws

moe-cli-proxy-dr:
  expose:
    - TLS/WSS endpoint
```

优点：

```yaml
pros:
  - 无 mixed content
  - 不依赖 loopback/socat
```

缺点：

```yaml
cons:
  - 需要域名/TLS/反代
  - 配置成本更高
```

当前场景不建议优先走 B。

# 6. O(1) 排障指导

## Probe 1：确认 aistudio-shard-0 内监听端口

交给 L3 Debug 执行：

```text
目标:
  判断容器内是否监听 7860/8317。

预期:
  - 7860 存在: Flask 正常。
  - 8317 若不存在: loopback WS 转发缺失，H1 成立。

结论格式:
  listen_7860=true/false
  listen_8317=true/false
```

## Probe 2：确认容器内能访问 moe-cli-proxy-dr

```text
目标:
  从 aistudio-shard-0 内访问 moe-cli-proxy-dr:8317/v1/models。

预期:
  HTTP 200 或有效 JSON。

结论格式:
  dns_moe_cli_proxy_dr=true/false
  http_8317_models=true/false
```

已有记录显示双向 DNS/网络大体通过，但还需要确认 `8317` 服务端口级可达。

## Probe 3：确认浏览器页面实际发起的 WS URL

利用现有 `WebSocketLogger`：

```text
操作:
  - 打开 /api/logs/ws/enable 或创建 ws log flag。
  - 重启 aistudio-shard-0。
  - 查看 logs/ws_messages 或 app.log 中 [WS] 新连接建立。

判定:
  - 出现 ws://127.0.0.1:8317/v1/ws: 页面配置正确，若失败则 socat 缺失。
  - 出现 ws://cliproxy-dr:8317/v1/ws: 可能 mixed content 或 DNS/NO_PROXY 问题。
  - 无任何 WS: AI Studio App 配置/执行异常，或页面未成功进入通信程序。
```

## Probe 4：检查 moe-cli-proxy-dr Provider 注册日志

```text
目标:
  找到 provider connected / disconnected / unauthorized / close 1006。

判定:
  - unauthorized: JWT_TOKEN/wsauth 错。
  - close 1006: 网络/心跳/页面保活问题。
  - no connection: WS 根本没到达 proxy。
```
